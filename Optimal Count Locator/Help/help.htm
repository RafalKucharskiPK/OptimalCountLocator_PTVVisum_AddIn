Optimal Count Locator

This add-on uses Visum Assignment results to determine most efficient set of Traffic Count Locations.
For more general info Visit i2 blogspot post regarding OCL

Install

To run application unzip attached .rar file into Visum_Installation_Folder/Exe/AddIns/
Restart Visum application (all opened Visum applications).
When Visum starts Add-On should be available via Menu->Scripts->Intelligent-Infrastructure->Optimal Count Locator
All modules needed to run script are by default installed with Visum. In case of any import problem try to re-install Visum with with full Python installation.

Input

To start working with Add-On you need to have network with traffic assignment opened in Visum.
Also make sure that you are using a method where paths are given explicitly (i.e. Equilibrium LUCE doesn't generate Visum PrT paths).
You can specify objects where Count Locations are already installed. Procedure generates UDA for links and nodes named: 'i2_OCL_Detectors'. Default value is 0, set it to 1 for object which are already fixed as count locations (OCL will set this value for 2 for selected optimal objects).

Setting Parameters:

You can set up to 6 parameters to handle the OCL properly:
a) Demand Segment - procedure works only for single demand segment. Choose proper DSeg from list.
b) Budget - specify how many objects do you want to count (integer).
c) Locate detectors on - you can tell application where should it locate detectors: 'links only', 'links and nodes' and 'nodes only' are available. Remember that links can be either directed, either undirected.
d) Optimization strategy - there several ways to reach optimum: either you catch as much flow as you can, either as much various OD pairs, either you mix it in bicriterial strategy - recommended to catch both volumes and OD pairs diversity.
e) Algorithm depth - parameter specifies how many Paths per OD pair is used. For procedures generating vast number of paths this parameter can speed up procedure significantly. Also if you want to focus on main paths only you can set value of 1 or 2 for this parameter (integer).
f) Undirected - you can determine detectors on link as one directional, or bidirectional respectively.

Calculation:

After parameters are specified you can press Calculate button.
Console in the middle of window will tell you what is being done at the moment.
Mind that gathering data about paths can take some time. It's about 100MB MS Access database for 100 000 paths, so it needs to take a while.
Also program may seem to have stopped after it downloaded paths - be patient - it's still working and after few seconds it will carry on.
When calculation is finished you can scroll console to see detailed results including statistics, etc.
Results are passed to Visum via UDA 'i2_OCL_Detectors'. OCL sets 2 as a value of parameter for resulting optimal count locations.
Full content of program console, with results can be found in program directory as report.txt file.

Graphical results:

We provided you with default .gpa files which can be opened via open GPA button, however in many cases this .gpa will not be feasible. In such case overwrite ShowFlow.gpa file in program directory.
show flow bundle button presents data as detected and undetected flows.

Main Nodes not supported:

if you have main nodes in your network OCL will not work. You need to remove all main nodes for purposes of calculations, afterwards you can easily transfer results for version file with main zones. OCL supporting main nodes available on request.

Trouble shooting:

With any problems don't hesitate to write to us at info@intelligent-infrastructure.eu. Please provide us with description of the error, and screenshot of window popping-up ("Python Error", or similar).
Please read the content of error message. If there is a problem related to module import, you can try to Google name of unimported module (i.e. Scipy, Numpy, Matplotlib, SQlite3) and install them using windows installer package (mind that up to Visum 12, python version 2.5 is installed with Visum).

32/64 bit versions

OCL works fine with both versions, however there are major performance issues while using 32 bit Visum on 64 bit machines. Computation times are rising significantly (i.e. from 2 minutes to 48 hours). For addressing such cases contact us - we have solutions.
